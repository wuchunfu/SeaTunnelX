# SeaTunnelX 告警中心：触发、恢复、关闭与通知说明

> 更新时间：2026-03-10  
> 目标：把告警中心收敛成用户容易理解的一套产品语义，而不是“状态 / 进度 / recovered 类型”到处并列。

---

## 1. 现在用户只需要理解 3 个主状态

告警中心最终只保留 3 个主状态：

### 1）触发中
表示问题现在还存在。

例子：
- 节点离线
- Agent 心跳超时
- CPU 持续过高
- 内存持续过高
- failed jobs 持续上升

### 2）已恢复
表示系统已经判断问题消失。

例子：
- 离线节点重新恢复在线
- Agent 心跳恢复
- CPU / 内存指标回落到阈值以内
- Prometheus / Alertmanager 把告警判定为 resolved

### 3）已关闭
表示这次告警事件已经被人工收尾，不需要继续留在“待关注”视角里。

说明：
- “已关闭”是对**这一次告警事件**的收尾；
- 不是永久关闭这个规则；
- 下次同类问题再次发生时，仍然会产生新的告警。

---

## 2. 为什么不再把“处理进度”做成主概念

之前容易让人困惑的是：
- 告警状态
- 处理进度
- 已确认
- 已静默
- 已恢复

这些概念并列以后，用户很容易分不清：
- 已确认是不是等于已恢复？
- 已静默是不是等于已处理？
- 已恢复是不是还要我确认？

所以现在产品收敛为：

### 主状态只看 3 个
- 触发中
- 已恢复
- 已关闭

### 其他动作降级成辅助标记
- 已确认：表示有人已查看
- 已静默：表示暂时不再打扰

它们仍然有价值，但不再和主状态平级展示。

---

## 3. 恢复不是一堆 `xxx_recovered` 产品类型

产品层面不会让用户看到很多碎片化概念，比如：
- agent_recovered
- master_recovered
- worker_recovered
- cpu_recovered
- memory_recovered

这些在产品上都不应该变成独立“状态体系”。

### 正确做法
恢复是**一条告警实例的生命周期变化**。

也就是说：
- 底层可以有很多事件；
- 上层统一成一条告警；
- 这条告警只会经历：
  - 触发中
  - 已恢复
  - 已关闭

---

## 4. master / worker / agent 怎么看待

这些不是独立的“恢复类型”，而是**对象标签**。

例如：
- 对象类型：节点异常
- 角色标签：master / worker
- 当前状态：触发中 / 已恢复 / 已关闭

所以：
- 我们不需要做“master recovered”这种用户概念；
- 只需要在告警详情里标明这个对象是 master 还是 worker。

### Agent 也是一样
Agent 的恢复不是单独一套产品语言，而是：
- 对象：Agent / Host / 节点链路
- 状态：触发中 / 已恢复 / 已关闭

---

## 5. 本地告警和 Prometheus 告警如何统一

对用户来说，它们是一套统一告警产品。

### 5.1 本地事件类告警
由 SeaTunnelX 自己判断触发与恢复。

例如：
- 节点离线
- 节点进程停止
- Agent 失联
- 本地运行态异常

恢复条件通常来自：
- 节点重新在线
- 进程重新启动
- Agent 心跳恢复
- 本地健康检查恢复正常

### 5.2 指标类告警
由 Prometheus / Alertmanager 提供更深层指标能力。

例如：
- CPU 持续高于阈值
- 内存持续高于阈值
- FD 使用率过高
- failed jobs 持续增加
- 线程池积压
- split-brain risk

恢复条件通常来自：
- PromQL 不再满足告警条件；
- Alertmanager 把状态从 firing 变成 resolved。

### 5.3 统一后的用户感知
用户不需要区分“本地恢复”还是“Prometheus 恢复”。

在告警中心里，他只看到：
- 当前状态
- 触发原因
- 恢复时间
- 通知记录

---

## 6. SeaTunnelX 当前推荐的恢复设计

### 6.1 节点 / 进程类
适合做恢复通知。

例子：
- 节点离线 -> 节点恢复在线
- 进程停止 -> 进程恢复运行

### 6.2 Agent 类
适合做恢复，但级别通常不应压过节点健康本身。

建议：
- 默认级别低于节点离线；
- 主要用于“控制面观测能力恢复”的提醒。

### 6.3 指标类
适合做恢复，而且恢复来源应直接复用 Prometheus / Alertmanager 的 resolved。

这不是“妥协”，而是标准设计：
- 指标表达式由 Prometheus 最懂；
- 展示与通知由 SeaTunnelX 统一承接。

---

## 7. 哪些应该进“告警生命周期”，哪些只记事件日志

### 应进入告警生命周期的
这类事件代表“异常条件”：
- 节点离线
- Agent 失联
- CPU / 内存 / FD 过高
- failed jobs 过多
- split-brain risk

它们应该有：
- 触发中
- 已恢复
- 已关闭

### 不应进入告警生命周期的
这类只是操作记录或信息事件：
- restart requested
- stop requested
- start requested
- 配置保存成功
- 通道测试成功

这类更适合：
- 事件日志
- 操作审计
- 历史记录

而不是进入统一告警生命周期。

---

## 8. 通知怎么发

### 8.1 触发通知
当问题第一次被确认触发时：
- 发送告警通知
- 邮件标题使用 `[ALERT]`
- 邮件内容突出当前问题和影响对象

### 8.2 恢复通知
当问题被确认恢复时：
- 可选择发送恢复通知
- 邮件标题使用 `[RECOVERED]`
- 邮件内容强调“该问题已恢复”

### 8.3 关闭动作
关闭是人工动作。

默认建议：
- 关闭本身不再额外发邮件；
- 它主要用于告警列表收尾与归档。

---

## 9. 已确认、已静默现在是什么角色

### 已确认
表示：
- 有人已经看到这条告警了；
- 但不代表问题已恢复。

### 已静默
表示：
- 短时间内不想继续被这条告警打扰；
- 但不代表问题已恢复；
- 也不代表这条告警事件已经关闭。

所以这两个现在都只是**辅助标记**，不是主状态。

---

## 10. 一个完整例子

### 例子 A：节点离线后又恢复
1. worker 节点离线
2. 告警状态 = 触发中
3. 发送告警邮件
4. 节点重新上线
5. 告警状态 = 已恢复
6. 发送恢复邮件（如果策略开启）
7. 运维确认无需继续跟进
8. 人工关闭
9. 告警状态 = 已关闭

### 例子 B：CPU 过高后回落
1. Prometheus 发现 CPU > 80% 持续 5 分钟
2. 告警状态 = 触发中
3. 发送告警邮件
4. CPU 降回阈值内
5. Alertmanager 判定 resolved
6. 告警状态 = 已恢复
7. 运维如需收尾，可人工关闭
8. 告警状态 = 已关闭

---

## 11. 当前实现约定

告警中心当前已经按下面的方向收敛：

### 列表主状态
- 触发中
- 已恢复
- 已关闭

### 辅助动作
- 确认
- 静默 30 分钟
- 对已恢复告警执行关闭

### 展示方式
- 主状态作为主列展示
- “已确认 / 已静默”只作为辅助标记展示
- 不再把“处理进度”单独做成主过滤维度

---

## 12. 一句话总结

> SeaTunnelX 告警中心现在遵循的是：**底层事件可以很多，上层状态只保留 3 个：触发中、已恢复、已关闭。**

这样做的好处是：
- 用户更容易理解；
- 本地告警和 Prometheus 告警可以统一；
- 恢复语义更自然；
- 不需要把 master / worker / agent / cpu / memory 各自再拆成一堆 recovered 类型。
