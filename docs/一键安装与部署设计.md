## SeaTunnelX 一键安装与部署设计（后端视角）

本设计文档面向开发与运维（而非最终用户），用于约束 **一键安装脚本** 与 **部署形态（单二进制 / Docker / Kubernetes）** 的整体方案和边界。

---

### 1. 设计目标与范围

- **统一体验**：无论是二进制、Docker 还是 Kubernetes，用户都应感知为“按照少量参数，一键拉起整套 Seatunnel 管控面 + （可选）观测栈”。
- **清晰分层**：
  - SeatunnelX 核心进程：后端 API、Frontend、Scheduler、Worker、Agent；
  - 可观测性栈：Prometheus / Grafana / Alertmanager（优先推荐接入已有远程监控栈，内置仅用于 demo）。
- **最小必需原则**：生产推荐路径只围绕 SeatunnelX 自身 + 与“远程监控栈”的集成；内置 Prometheus/Grafana/Alertmanager 仅作为本地体验或 demo。
- **易于升级与回滚**：一键脚本需要考虑版本化、幂等性和回滚路径，避免“脚本执行一次就无法重来”的情况。

本文不深入到具体脚本实现细节（bash / PowerShell / Helm Template 等），重点聚焦“需要考虑什么”和“各形态的统一约束”。

---

### 2. 组件与进程视角

#### 2.1 核心组件

- **Control Plane Backend**（后端 API + gRPC）：
  - 二进制：`seatunnelX api`
  - Docker：`seatunnelx-backend` 镜像
  - K8s：`Deployment` / `StatefulSet`
- **Frontend**（Next.js）：
  - 二进制形态通常由 Node 进程或静态文件 + Nginx 提供
  - 也可构建为独立前端镜像
- **Scheduler / Worker**：
  - 当前仅保留 legacy 命令壳
  - 默认部署不再依赖其能力，也不再要求 Redis Broker
- **SeatunnelX Agent**：
  - 部署在目标主机，单独的安装与运维路径（本设计文档不展开）。

#### 2.2 可观测性组件（可选）

推荐优先对接已有 Prometheus/Grafana/Alertmanager 集群，SeaTunnelX 只负责：

- 暴露 HTTP SD 接口；
- 提供 Alertmanager Webhook；
- 提供 Grafana Dashboard JSON；

内置三件套仅用于本地开发 / demo，可通过单独脚本或 Docker Compose 一键拉起，不纳入生产推荐链路。

---

### 3. 一键安装脚本的通用设计原则

无论是“单机二进制”、“Docker Compose”还是“Kubernetes/Helm”，一键安装脚本应遵循以下通用原则：

- **参数最小化**：
  - 对最终用户暴露的参数尽量少，核心仅包括：
    - 访问域名 / IP（用于 `app.external_url`）；
    - 监听端口（HTTP / gRPC）；
    - 数据库类型和连接信息（可默认 SQLite）。
- **配置模板化**：
  - 所有部署形态均从同一份 `config.example.yaml` 模板（或其子集）生成最终 `config.yaml`；
  - 变更参数只需写入少数键值，剩余使用合理默认。
- **幂等性**：
  - 多次执行一键脚本，不应重复创建冲突资源：
    - 已存在的目录 / 配置文件应提示并支持“覆盖 / 跳过 / 备份再覆盖”策略；
    - 系统服务 / K8s 资源 / Docker 容器存在时，应优先走“升级重启”路径。
- **可观测性集成清晰**：
  - 生产路径：默认只启用远程集成（不自行拉起 Prometheus/Grafana/Alertmanager）；
  - 本地体验路径：提供额外参数或脚本（如 `--with-demo-observability`）来拉起本地监控三件套。
- **安全默认值**：
  - 强制要求用户修改初始管理员密码；
  - 在对公网暴露时推荐前置 HTTPS 反向代理；
  - 对外端口最小化暴露（例如 gRPC 仅对内部网络开放）。

### 4. 单二进制部署（一键安装考虑点）

#### 4.1 目录与文件布局

推荐目标目录结构（以 Linux 为例）：

```text
/opt/seatunnelx/
  bin/
    seatunnelX              # 后端主二进制
  config/
    config.yaml
  data/
    db/                     # SQLite 或其它本地数据
    storage/                # 安装包、插件等
  logs/
    seatunnelx.log
  deps/
    grafana_config/         # Dashboard JSON 等静态资源
```

一键安装脚本需要完成：

- 创建上述目录结构；
- 将发布包中的二进制、模板配置、文档拷贝到位；
- 根据用户输入（或交互式问答）生成最终 `config.yaml`。

#### 4.2 系统服务与进程管理

在生产场景，推荐注册为系统服务（以 systemd 为例）：

- `seatunnelx-api.service`：`seatunnelX api`（后端主进程）
- （可选）`seatunnelx-frontend.service`：前端 Node `server.js` 进程，见 4.3.3

一键脚本需要：

- 检查当前系统是否支持 systemd；
- 生成对应的 service 单元文件（指定 `WorkingDirectory`、`ExecStart`、`Restart=always`、`LimitNOFILE` 等）；
- 启动并校验服务是否成功（健康探活 / 日志检查）。

#### 4.3 前端部署考虑（Next.js standalone + Node）

当前推荐的前端运行模式是 **Next.js standalone 模式 + Node `server.js` 启动**：

- 编译阶段将 Next.js 项目打包为 standalone 产物（含 `server.js` 等运行时代码）；
- 运行阶段通过内置的 Node 二进制执行，例如：
  - `node server.js` 或 `./node server.js`。

为保证在不同 Linux 发行版上的兼容性与可移植性，一键安装脚本与打包过程需要关注以下几点。

##### 4.3.1 Node 运行时与安装包划分

前端运行依赖 Node 运行时，但为了简化一键脚本逻辑，同时兼容 CentOS 7 等低版本 glibc 的环境，**安装包层面直接区分两套发行物**：

- **标准安装包（glibc ≥ 2.27 环境）**：
  - 内置官方 Node 22（或后续官方 LTS）二进制；
  - 适用于主流现代 Linux 发行版。
- **兼容安装包（CentOS 7 等低 glibc 环境）**：
  - 内置来自 Node.js “unofficial-builds” 子项目的兼容构建；
  - 下载来源示例：[`https://unofficial-builds.nodejs.org/download/release/`](https://unofficial-builds.nodejs.org/download/release/)；
  - 虽为“不担责的官方编译版”，但由 Node.js 官方项目维护，托管在 `nodejs.org` 域名下。

两套安装包的目录结构保持完全一致，例如：

```text
/opt/seatunnelx/
  bin/
    seatunnelX
  frontend/
    server.js           # Next.js standalone 启动入口
    .next/standalone    # Next.js 打包产物
  deps/
    node/
      node              # 当前安装包自带的 Node 二进制
```

区别仅在于：**标准包与兼容包中 `deps/node/node` 的编译来源不同**。

##### 4.3.2 glibc 版本探测与用户提示（仅二进制一键脚本）

一键安装脚本（仅针对二进制部署形态）需要在安装早期阶段完成以下逻辑：

- 读取系统 glibc 版本（例如通过 `ldd --version` 或其它方式），与当前安装包自带 Node 的最低要求进行对比；
- 行为约定：
  - 若当前系统 glibc 版本 **满足** 本安装包要求：
    - 继续安装流程，无需额外提示；
  - 若当前系统 glibc 版本 **低于** 本安装包要求（例如使用标准包安装到 CentOS 7）：
    - 终止安装流程，并给出明确错误提示：
      - 当前 glibc 版本；
      - 当前安装包类型（标准 / 兼容）；
      - 引导用户下载对应的“兼容安装包”或指定版本安装包链接。

重要的是：

- 一键脚本 **不在同一安装包内动态切换 Node 版本**，也不尝试自动下载其它包；
- 兼容性由“多安装包策略”保证，脚本只负责做**一次性的环境检测和友好的错误提示**。

##### 4.3.3 前端进程管理

无论是使用官方 Node 还是 unofficial-builds Node：

- 一键脚本可以为前端生成独立的 systemd service（`seatunnelx-frontend.service`），或将前端集成进统一 service：
  - `ExecStart` 形如：`/opt/seatunnelx/deps/node/<platform>/node /opt/seatunnelx/frontend/server.js`；
- 确保：
  - 日志输出路径规划清晰（可写入统一 `logs/` 目录）；
  - 端口占用与后端 API 协调（前端通常监听 3000 或由反向代理统一入口）。

---

#### 4.5 安装包形态与架构矩阵（含 Agent）

针对一键安装（单机二进制部署）场景，发布物需要同时考虑：

- CPU 架构：`amd64` 与 `arm64`；
- Node 运行时对 glibc 版本的要求（控制面 `seatunnelX` 与 `seatunnelx-agent` 本身为**纯静态编译**，不依赖系统 glibc 动态库，glibc 兼容性问题只影响 Node）。

构建阶段固定产出 **4 个安装包**，文件名同时编码架构与 Node 适配的 glibc 版本（便于运维直接看包名判断适用环境）：

```text
seatunnelx-{version}-linux-amd64.tar.gz   # amd64，正常发行版
seatunnelx-{version}-linux-amd64-glibc217.tar.gz   # amd64，适配 glibc 2.17（如 CentOS 7）等环境
seatunnelx-{version}-linux-arm64.tar.gz   # arm64，正常发行版
seatunnelx-{version}-linux-arm64-glibc217.tar.gz   # arm64，适配 glibc 2.17 等环境
```

说明：`glibc217` 仅表示本包内置的 **Node 运行时** 所针对的最低 glibc 版本；`seatunnelX` 与 `seatunnelx-agent` 仍保持静态编译。

每个安装包内部的目录布局保持一致，例如：

```text
/opt/seatunnelx/
  bin/
    seatunnelX                          # 控制面主二进制（静态编译，按架构区分）
  lib/
    seatunnelx-java-proxy-2.3.13.jar   # 默认 runtime probe 薄 jar（未来可并存多版本）
    agent/
    seatunnelx-agent-linux-amd64        # Agent 二进制（静态编译，按架构后缀区分）
    # 或 seatunnelx-agent-linux-arm64
  scripts/
    seatunnelx-java-proxy.sh       # Agent 安装脚本分发给目标主机的 runtime probe 启动脚本
  frontend/
    server.js                           # Next.js standalone 入口
    .next/standalone                    # 前端打包产物
  deps/
    node/
      node                              # 本包内置的 Node 运行时（官方版或 unofficial-builds 版）
  config/
    config.yaml                         # 初始配置（或模板）
  docs/
    ...                                 # 文档
```

说明：

- `seatunnelX` 与 `seatunnelx-agent-*` 采用静态编译方式，避免对系统 glibc 版本产生硬依赖；
- `seatunnelx-java-proxy-{seatunnelVersion}.jar` 与 `scripts/seatunnelx-java-proxy.sh` 作为 Agent 的辅助安装资产统一放入发布包，供远端 runtime storage probe 使用；
- Agent 运行时优先按 SeaTunnel 集群版本选择 `seatunnelx-java-proxy-{seatunnelVersion}.jar`，找不到时回退到 `seatunnelx-java-proxy-2.3.13.jar`；
- Node 运行时依旧可能对 glibc 有最低版本要求，因此：
  - 构建流水线可以针对不同目标环境（例如“现代发行版 / CentOS7 兼容”）产出不同 Node 变体，对应不同的下载链接，但目录结构完全一致；
  - 一键安装脚本在运行时只面对“当前安装包”：
    - 通过 glibc 检测判断当前系统是否满足本包内 Node 的最低要求；
    - 若不满足，则中止安装并提示用户“请下载适配当前环境的安装包”（例如 CentOS7 专用包），而不是在脚本中动态切换二进制或下载新包。

#### 4.4 可观测性集成

一键脚本在生成 `config.yaml` 时，需要：

- 写入正确的 `app.external_url`（影响 HTTP SD、Alertmanager Webhook 等路径）；
- 若用户勾选“启用远程监控集成”，则提示输入：
  - `observability.prometheus.url`
  - `observability.alertmanager.url`
  - `observability.grafana.url`

此外，可在完成安装后输出 Prometheus / Alertmanager / Grafana 的一次性配置片段，方便运维复制到对应系统。

### 5. Docker / Docker Compose 部署（一键安装考虑点）

#### 5.1 镜像与服务拆分

容器化部署我们提供两条主路径：

- **双镜像模型（生产推荐，配合 docker-compose）**
- **单镜像模型（前后端合并，单条 `docker run` 极速体验）**

##### 5.1.1 双镜像模型（生产推荐）

SeatunnelX 的标准容器化设计以 **前后端两种业务镜像** 为核心：

- `seatunnelx-backend`：后端 API + gRPC；
- `seatunnelx-frontend`：前端应用（Next.js standalone + Node `server.js`）。

说明：

- Scheduler / Worker 的运行方式由后端程序内部控制，在镜像与部署层面不再额外抽象为独立服务。
- 可观测性相关镜像（如 `prom/prometheus`、`grafana/grafana`、`prom/alertmanager`）是否拉起，取决于用户是否在 SeatunnelX 中开启可观测性集成：
  - **建议生产环境开启可观测性**，但优先对接用户已有的集中式监控栈；
  - Docker 形态下，仅在用户选择“本地 demo 监控栈”或测试场景时，才通过额外容器拉起上述镜像。

##### 5.1.2 单镜像模型（前后端合并，极速体验）

为支持“一条命令体验 SeatunnelX”，可以在构建阶段额外提供一个**前后端合并镜像**：

- `seatunnelx-all-in-one`：
  - 内部同时包含 backend 与 frontend 运行时；
  - 默认使用 SQLite 作为后端数据库；
  - 容器内通过 `supervisor` / `s6` / 自研多进程管理等方式同时拉起后端 API 与前端服务；
  - 适合作为本地试用、PoC 或 CI 环境的快速启动方式。

该镜像不承担生产级监控/扩缩容职责，生产环境仍推荐使用 `seatunnelx-backend` + `seatunnelx-frontend` 双镜像配合 docker-compose / K8s 进行更细粒度的资源和生命周期管理。
#### 5.2 docker-compose.yml 设计

一键安装脚本在 Docker 场景下应该支持两种模式：

- **模式 A：docker-compose + 双镜像（生产推荐）**
- **模式 B：单镜像 `docker run` 极速体验**

模式 A（docker-compose，生产推荐）：

- 至少包含：
  - `seatunnelx-backend`（后端 API/gRPC）；
  - `seatunnelx-frontend`（前端服务）；
  - 可选 MySQL/PostgreSQL（若不使用则退回 SQLite）；
- 根据用户是否开启可观测性：
  - 若对接外部监控栈：不在 Compose 中拉起 Prometheus / Grafana / Alertmanager，仅通过 `config.yaml` 中的 `observability.*.url` 进行集成；
  - 若用户选择“本地 demo 监控栈”：在 Compose 中额外添加 `prom/prometheus`、`grafana/grafana`、`prom/alertmanager` 三个服务及对应卷。

模式 B（单镜像 `docker run`，极速体验）：

- 使用 `seatunnelx-all-in-one` 镜像，典型启动命令示例：
  - `docker run -p 8000:8000 -p 3000:3000 seatunnelx-all-in-one:{version}`
- 特点：
  - 内部自带 SQLite 与前后端进程管理；
  - 不强依赖外部数据库或监控栈；
  - 主要用于本地快速体验和 demo，不作为生产环境的首选部署方式。

一键安装脚本在输出中应同时给出：

- 一份 `docker-compose.yml` 示例（模式 A）；
- 一条 `docker run` 示例命令（模式 B），便于用户在本机快速拉起 all-in-one 容器。

关键考虑点：

- 配置注入：
  - 通过环境变量或挂载 `config.yaml` 方式将配置注入后端；
  - 保证 `app.external_url` 和数据库配置与外部实际环境一致。
- 数据卷管理：
  - `data/`、`logs/` 等目录需挂载为持久卷；
  - 如果使用 MySQL/PostgreSQL，需单独的持久化卷映射。
- 网络与安全：
  - Compose 网络中对外暴露的端口最小化，如仅暴露前端 HTTP 和必要的 API；
  - gRPC 端口可以仅开放在内部网络中。

#### 5.3 一键脚本职责（Docker 形态）

一键脚本需要：

- 询问或读取必要参数（如域名、端口、数据库连接信息）；
- 在目标目录生成 `docker-compose.yml` 和基础 `.env` 文件；
- 拉取必要镜像（`docker compose pull`）；
- 启动并检查服务健康（`docker compose up -d` 后探活）。

### 6. Kubernetes / Helm 部署（一键安装考虑点）

Kubernetes 场景下推荐使用 Helm Chart 或 Kustomize，以 Helm 为主说明。

#### 6.1 Chart 结构与资源划分

建议的 Helm 结构：

```text
charts/seatunnelx/
  Chart.yaml
  values.yaml
  templates/
    backend-deployment.yaml
    backend-service.yaml
    frontend-deployment.yaml
    frontend-service.yaml
    configmap.yaml
    secret.yaml
    ingress.yaml
```

关键资源考虑：

- `Deployment`：
  - 后端、前端、Scheduler、Worker 各自的副本数、扩缩容策略；
- `Service`：
  - 对外暴露 HTTP / gRPC 端口；
  - 内部服务通信；
- `Ingress`：
  - 提供 HTTP(S) 域名访问，配合 TLS 证书；
  - 影响 `app.external_url` 的值。
- 配置与密钥：
  - 使用 `ConfigMap` 注入 `config.yaml` 或核心配置信息；
  - 使用 `Secret` 存储数据库密码、管理员密码等敏感信息。

#### 6.2 监控集成（Kubernetes）

生产推荐依旧是对接现有 Prometheus / Alertmanager / Grafana：

- Prometheus Operator 场景下：
  - 可以通过 `ServiceMonitor` / `PodMonitor` 发现 SeatunnelX 自身 metrics；
  - 同时仍然使用 HTTP SD 方式为 Seatunnel 集群提供统一的 metrics endpoint 列表；
- 一键部署不负责创建监控栈，只负责：
  - 保证 `observability.enabled` 配置正确；
  - 暴露 HTTP SD 与 Alertmanager Webhook 的 Ingress 规则；
  - 可选：在 Helm `values.yaml` 中提供 Prometheus / Alertmanager / Grafana 的 URL 配置项。

#### 6.3 Helm 一键安装的考虑点

- `values.yaml` 中需要暴露的关键项：
  - `image.repository` / `image.tag`；
  - `ingress.enabled` / `ingress.hosts` / `ingress.tls`；
  - `resources`（CPU/内存 request/limit）；
  - `persistence`（数据卷大小、StorageClass）；
  - `observability.enabled` 与相关 URL。
- 一键脚本可以封装为：

```bash
helm repo add seatunnelx ...
helm install seatunnelx seatunnelx/seatunnelx \
  --set ingress.hosts[0].host=<domain> \
  --set app.externalUrl=https://<domain> \
  --set observability.enabled=true \
  ...
```

脚本职责：

- 校验 `kubectl` / `helm` 是否可用；
- 根据用户输入生成临时 `values-override.yaml`；
- 调用 `helm install` 或 `helm upgrade --install` 完成部署；
- 输出访问地址和默认管理员账号信息。

### 7. 多形态统一的关键关注点

无论采用哪种部署方式，一键安装与后续运维都需要关注以下统一点：

- **配置一致性**：
  - 所有形态最终都应落在同一套配置模型上（以 `config.yaml` 为中心），避免“二进制 / Docker / K8s 三套完全不同配置”的局面。
- **升级与回滚**：
  - 一键脚本应该明确升级路径（如何备份旧版本、如何恢复）；
  - 在二进制形态下，至少备份旧二进制与 `config.yaml`，在 Docker/K8s 形态下保留旧镜像 tag 和旧配置版本。
- **日志与观测**：
  - 统一日志路径和格式，便于在不同部署形态下接入日志系统；
  - 保证 HTTP SD 与 Alertmanager Webhook 在三种形态下地址一致（通过 `app.external_url` 和 Ingress/反向代理统一）。
- **安全**：
  - 强制初始化管理员密码；
  - 清晰区分“内网端口”和“对公网暴露端口”；
  - 推荐使用 HTTPS 入口，并在文档中标注 HTTP 仅用于 PoC/内网。

### 8. GitHub Actions 发布与 CI 策略（落地）

为配合一键安装包与镜像发布，仓库侧约定两条工作流：

1. **主分支 CI（`main`）**
   - 触发条件：
     - `push` 到 `main`（包含 PR 合并后）；
     - `pull_request` 指向 `main`。
   - 目标：
     - 后端/Agent 执行 `go test`；
     - 前端执行 `pnpm test` 与 `pnpm build`；
     - 在代码合并前后提供自动化质量门禁。

2. **Tag 发布（`v*`）**
   - 触发条件：`push` tag（如 `v1.2.0`）。
   - 目标：
     - 调用 `scripts/package-release.sh` 产出一键安装包；
     - 以 **Node22 为主矩阵** 产出 `official` 与 `glibc217` 两个变体；
     - 每个变体同时生成 `with-observability` / `without-observability` 包；
     - 上传产物并附加到 GitHub Release；
     - 构建并推送镜像到 **GHCR**（`ghcr.io/<owner>/...`）。

镜像仓库策略：

- 统一使用 GitHub Container Registry（GHCR）；
- 至少发布三类镜像：
  - `seatunnelx-backend`
  - `seatunnelx-frontend`
  - `seatunnelx-all-in-one`（前后端合并，适合一键体验）
- Tag 发布镜像标签与 git tag 保持一致，便于回滚与审计。

### 9. 结语

本设计文档定义了 SeaTunnelX 在 **单二进制 / Docker / Kubernetes** 三种形态下“一键安装”需要考虑的核心点与边界，目标是：

- 避免在不同部署方式之间出现行为和配置模型的割裂；
- 为后续脚本实现（安装脚本、Compose 模板、Helm Chart）提供统一的设计约束；
- 为运维人员提供一个“从单机到集群”的平滑迁移路径。

在实际落地时，可以将本设计与《00-快速开始》和《observability-remote-design》结合使用：  
前者面向最终用户“如何启动”，后者定义可观测性集成，本设计主要定义“整体部署与一键安装的形态与边界”。
