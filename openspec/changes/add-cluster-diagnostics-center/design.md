## Context

SeaTunnelX 当前已经具备多块与诊断相关但彼此分散的能力：

- 集群详情页已经能查看节点、进程状态、升级记录、监控配置与进程事件；
- 监控中心已经有告警中心、策略中心、平台健康与集群健康；
- Agent 已深度集成 Seatunnel 进程，持有 `InstallDir`、`Role` 等上下文；
- gRPC 已有 `LogStream` 入口，Agent 协议已存在 `COLLECT_LOGS`、`THREAD_DUMP`、`JVM_DUMP` 诊断命令；
- 升级中心已经沉淀了一套成熟的“任务 / 步骤 / 节点执行 / 日志 / 事件流”交互模式。

当前缺失的是一个统一的“诊断中心”域：错误日志收集仍停留在按节点读取单个日志文件；巡检与诊断包尚无统一的数据模型和产品入口；告警、集群详情、错误排查之间无法形成稳定闭环。与此同时，后续 AI 分析需要结构化、可复用的诊断证据，而不是原始全量日志。

约束如下：

- 主链路依赖 Agent，不以 SeaTunnel REST API 为核心依赖；
- 初版错误采集仅关注 `ERROR`，不落全量原始日志；
- 必须兼容 `routingAppender`，因此不能只读取 `seatunnel-engine-worker.log`；
- 不引入新的外部日志中间件；
- 需要与当前监控、告警、升级执行等现有实现对齐，避免重复造轮子。

## Goals / Non-Goals

**Goals:**
- 提供统一的诊断中心入口，承载错误中心、巡检、诊断包三块能力。
- 建立以 Agent 为执行面的错误采集与诊断证据沉淀机制。
- 复用现有进程监控、告警、升级任务执行模型，降低新增系统复杂度。
- 建立跨页面联动：集群详情、告警中心、错误详情、巡检结果、诊断包任务之间可互相跳转。
- 为后续 AI 分析提供结构化输入，包括错误组、巡检发现、诊断任务与产物清单。

**Non-Goals:**
- 不建设通用全文日志平台，也不保存 Seatunnel 全量原始日志。
- 不在本变更中实现 AI 自动分析或自动修复。
- 不引入 Elastic、Loki、Kafka 等额外组件。
- 不在初版实现复杂的定时编排平台；巡检与诊断任务先支持手动触发，再为后续调度留扩展点。

## Decisions

### Decision 1: 诊断能力作为独立产品域建设，而不是继续堆叠到监控中心
- 选择：新增“诊断中心”工作台与独立后端领域模块，监控中心继续聚焦告警/策略/可观测性。
- 原因：错误中心、巡检、诊断包虽然与监控强相关，但交互模式更偏“问题排查与证据编排”。继续堆叠到 `/monitoring` 会让告警中心与诊断任务混杂，边界越来越模糊。
- 备选方案：直接在现有 MonitoringCenter 增加 tabs。
- 不选原因：会把观测态与诊断态强耦合，难以承载后续诊断包任务流与 AI 分析入口。

### Decision 2: 错误中心主链路采用 Agent 文件扫描 + LogStream 分流
- 选择：Agent 直接扫描 `${installDir}/logs` 下的 `seatunnel-engine-*.log` 与 `job-*.log`，识别 ERROR 事件后通过现有 `LogStream` 上报；Control Plane 在 `LogStream` 处理器中根据 source/type 分流进入新的错误域，而不是继续只落 audit log。
- 原因：Agent 已掌握 installDir/role，上报链路已存在；直接文件扫描天然兼容 routingAppender，且不依赖 Seatunnel API。
- 备选方案：继续复用 `get_logs` 拉取式接口，或直接读取 Seatunnel REST/metrics。
- 不选原因：`get_logs` 只适合人工补查；REST/metrics 既不能覆盖 `job-*.log`，也不适合后续 AI 证据沉淀。

### Decision 3: 错误数据模型只保留结构化证据，不存全量原文
- 选择：定义 `error_event`、`error_group`、`log_cursor` 作为最小模型，并为事件保留有限上下文、堆栈摘要、来源文件、jobId、cluster/node 关联。
- 原因：用户已明确只采 ERROR，不做全量日志平台；结构化模型更适合去重、查询、聚合与 AI 分析。
- 备选方案：保留原始完整日志并异步分析。
- 不选原因：存储成本高、性能复杂度高，且不符合“无额外组件”的约束。

### Decision 4: 巡检与诊断包复用升级中心的任务执行范式
- 选择：诊断任务（尤其诊断包）采用与升级中心一致的“任务 / 步骤 / 节点执行 / 日志 / SSE 实时流”模式；巡检初版可先产出报告模型，但执行与展示也沿用该范式，避免未来二次重构。
- 原因：仓库里 `stupgrade` 已有成熟实现，适合作为诊断域的任务执行骨架。
- 备选方案：巡检做同步接口、诊断包再单独做一套异步任务系统。
- 不选原因：会产生两套执行与日志模型，后续 AI、前端交互、状态追踪都更难统一。

### Decision 5: 诊断包以“来源驱动”建模，并复用现有诊断命令
- 选择：诊断包任务支持 manual / error_group / inspection_finding / alert 等触发来源；执行步骤复用已有 `COLLECT_LOGS`、`THREAD_DUMP`、`JVM_DUMP` 命令，并补充配置快照、进程事件与告警快照采集。
- 原因：这能让错误、巡检、告警形成闭环，并最大限度利用当前 Agent 协议。
- 备选方案：先只支持手动打包。
- 不选原因：会导致后续从错误中心、巡检结果跳转生成诊断包时又要重新改模型。

### Decision 5.1: MVP 阶段仅登记 JVM dump 元数据与远端路径，不立即回传 hprof 二进制
- 选择：当前 MVP 中，Agent 侧 JVM dump 成功后只回传结构化元数据（状态、大小、磁盘空间、远端输出路径），控制面 manifest 记录该远端产物；**hprof 二进制文件回传控制面**放到 5.5 / 后续增强阶段。
- 原因：hprof 文件通常较大，直接回传会放大控制面网络与存储压力，也会引入失败重试、断点续传、磁盘配额等额外复杂度。
- 备选方案：在 5.3 即完整上传二进制 hprof 到控制面。
- 不选原因：会显著扩大 MVP 实施范围，不利于先完成“可创建、可执行、可查看、可追溯”的主链路闭环。

### Decision 6: 前端采用“全局诊断中心 + 集群详情快捷入口”双入口模型
- 选择：新增 `/diagnostics` 工作台，默认展示错误中心；在集群详情页新增诊断概览卡片和快捷动作，在告警中心保留跳转到诊断中心的入口。
- 原因：全局入口适合跨集群排查热点，集群详情页适合就地闭环操作，两者缺一不可。
- 备选方案：只做一个全局页面，集群详情不嵌入。
- 不选原因：用户在集群排障场景下需要低跳转成本；仅有全局页会让日常运维动作割裂。

## Risks / Trade-offs

- [风险] 诊断域跨越 Agent、gRPC、后端、前端多个模块，变更面大。  
  → 缓解：按 M1 错误中心、M2 巡检、M3 诊断包分阶段推进，并优先复用现有模式。

- [风险] `LogStream` 目前主要用于 audit log，新增错误分流可能影响现有行为。  
  → 缓解：保持默认 audit 路径不变，仅当 `source=seatunnel_error` 等诊断标记存在时进入新链路。

- [风险] routingAppender 下日志格式与 jobId 提取不稳定。  
  → 缓解：jobId 采用“文件名优先、正文回退、无法识别则留空”的策略，并保证错误组聚合不依赖 jobId 必然存在。

- [风险] 诊断任务若过度抽象为通用任务引擎，会拖慢落地。  
  → 缓解：首版直接镜像 `stupgrade` 的任务模型到 diagnostics 域，待能力稳定后再抽公共层。

- [风险] 前端页面过多导致导航复杂。  
  → 缓解：以“诊断中心”统一承载三块能力，集群页只放摘要与快捷入口，不再零散新增页面。

## Migration Plan

1. 新增 OpenSpec 规格并按 M1/M2/M3 拆分实现。
2. M1 先落地错误中心的数据模型、Agent 错误采集、后端查询接口与基础页面。
3. 在错误中心稳定后，M2 接入巡检报告与发现项，并在集群详情页增加巡检入口。
4. M3 基于既有 Agent 命令与任务执行范式落地诊断包，并打通错误/巡检/告警来源联动。
5. 发布时保留原有节点日志查看与监控中心入口；新入口以增量方式加入，不破坏现有运维链路。
6. 如需回滚，可关闭诊断中心前端入口并停用错误采集上报，旧的日志补查、监控、告警能力不受影响。

## Open Questions

- 诊断包产物的最终存储介质是数据库登记 + 本地文件，还是直接复用现有 package/storage 目录约定，需要在实现前最终确认。
- 巡检的定时调度是否在 M2 即需要预留最小调度字段，还是完全留到后续变更。
- 错误组的指纹算法是否需要在初版支持规则可配置，还是先采用固定标准化算法。
