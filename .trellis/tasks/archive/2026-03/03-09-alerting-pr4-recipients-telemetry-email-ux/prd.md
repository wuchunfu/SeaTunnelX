# implement alerting PR-4 recipients, telemetry templates, and email UX

## Goal

把 SeaTunnelX 现有“告警策略 + 通知渠道”从已经可用的本地告警闭环，继续推进到更接近 DorisManager 的产品形态：

1. **补齐 Prometheus / SeaTunnel telemetry 驱动的内建策略模板**；
2. **把通知接收人从“通道配置”中抽离出来，与用户体系联动**；
3. **重做邮件通知配置与策略编辑体验**，让用户看到的是一套统一的告警产品，而不是“本地/远程”两套割裂能力；
4. **对不可用能力进行感知与隐藏**：Prometheus 未安装、未接入、指标不满足时，不向用户展示对应模板或入口。

---

## Requirements

### R1. 扩展内建告警模板

基于 SeaTunnel telemetry / Prometheus 能力补充模板库，至少覆盖以下类型：

- 节点或进程内存持续高于阈值（例如 80%）
- CPU 持续高负载
- 文件句柄 / FD 使用过高
- 失败作业数持续增加或超过阈值
- 线程池积压 / 拒绝
- 节点状态异常
- 集群不安全 / 脑裂风险（例如 cluster safe 相关指标）

要求：

- 模板要区分依赖能力：`telemetry`、`host_metrics`、`telemetry+host_metrics`
- **不允许伪造能力**：例如没有 host metrics 时，不能把 `process_cpu_seconds_total` 直接包装成“CPU 使用率 80%”
- 后端要根据当前接入能力决定模板可见性；前端只展示当前可用模板

### R2. 通知接收人与用户体系联动

策略的“通知对象”应基于系统用户，而不是仅靠 SMTP 配置里手填邮箱。

要求：

- 用户模型补充邮箱相关信息（至少支持 email）
- 提供“可通知用户”查询能力
- 告警策略支持选择多个接收人
- 默认选中管理员（admin）作为默认接收人
- 只有配置了有效邮箱的用户，才允许被加入邮件接收列表

### R3. 邮件通道只负责发信配置

邮件通知通道的职责应聚焦为 SMTP 发信配置，而不是同时承载业务收件人。

要求：

- 邮件通道保存：名称、描述、启用状态、服务器地址、端口、安全类型、用户名、密码、发件地址、发件人名称等
- 策略层保存：通知接收人（用户多选）
- 测试发送允许临时选择收件人，且应校验该用户是否已配置邮箱

### R4. 重做邮件通知配置 UI

邮件通知配置弹窗应采用主从布局，明显区分“通道管理”和“通道编辑”。

目标交互：

- 左侧：搜索、新建、通道列表
- 右侧：分区表单（基本信息 / 连接配置 / 发信配置 / 测试与状态）
- 顶部提供：测试连接、发送测试邮件
- 底部提供：删除、取消、保存草稿、保存并启用
- 显示状态：未测试 / 连接成功 / 发送失败 / 最近测试时间 / 错误详情
- 表单应具备实时校验、脏数据提醒、密码掩码、端口默认值联动

### R5. 告警策略编辑体验收敛

策略编辑体验应继续向 DorisManager 靠拢，只保留用户真正需要配置的内容。

要求：

- 作用集群必须返回真实的集群列表，不再显示 cluster id 原值
- 策略模板名称与说明支持中英文 i18n
- 策略编辑中可直接看到“通知对象”选择，而不是跳到通道里找收件人
- 用户感知上是一套统一告警能力；不要在主要交互中强调“本地告警 / 远程告警”概念

### R6. 跨层契约明确

本任务会改动数据库、后端 DTO / API、前端表单与服务层，必须明确边界契约：

- 用户邮箱字段的存储、校验、返回格式
- 策略接收人字段的持久化格式与 API 输出格式
- 模板可见性的计算规则与前端展示字段
- 邮件通道测试状态的存储与返回格式

## Acceptance Criteria

- [ ] 后端支持为用户维护邮箱信息，并能返回“可通知用户”列表
- [ ] 告警策略支持保存多个接收用户，并在通知时按用户邮箱解析收件人
- [ ] 默认管理员会出现在策略接收人默认值中
- [ ] 邮件通道配置与策略接收人职责分离，测试发送支持临时指定接收用户
- [ ] 前端策略模板列表只显示当前能力可支持的模板
- [ ] 至少补齐一组 Prometheus / telemetry 模板，并正确标记其依赖能力
- [ ] 邮件通道 UI 调整为主从布局，具备测试连接 / 测试发送 / 状态反馈
- [ ] 模板名称与描述具备中英文文案
- [ ] 集群选择使用真实集群列表数据，不再显示集群 ID
- [ ] 用户视角下，告警策略与通知流程呈现为一套统一产品体验

## Technical Notes

### Product decisions already agreed

- 对用户来说，告警能力应当是**统一产品**，不需要暴露“本地 / 远程”分裂概念
- Prometheus 是为了补足 SeaTunnel 集群内部指标、宿主机指标、失败作业等深度监控能力
- 如果 Prometheus / Alertmanager / 指标采集未就绪，则对应能力应自动隐藏或降级，而不是展示一个不可用入口
- 邮件通道负责“怎么发”，策略负责“发给谁”

### Expected backend changes

可能涉及但不限于：

- `auth_users` 增加邮箱字段及相关迁移
- monitoring policy 模型增加接收人字段（如 user IDs / normalized receivers）
- 新增“可通知用户”查询接口
- 新增 / 调整模板能力探测与模板列表接口
- 邮件通道增加测试状态 / 最近测试结果字段

### Expected frontend changes

可能涉及但不限于：

- Monitoring policy center 策略编辑表单
- 邮件通道配置弹窗重构
- admin user / user center 邮箱字段展示与编辑
- monitoring service types / i18n 文案更新

### Suggested implementation order

1. 用户邮箱字段 + 可通知用户接口
2. 策略接收人持久化与投递解析
3. 模板能力探测与 Prometheus / telemetry 模板扩展
4. 邮件通道 UI 重构与测试动作拆分
5. 前后端联调、重启、手工验证通知链路
