# design abnormal auto recovery feature and task flow

## Goal

为 SeaTunnelX 的“异常自动恢复”能力建立一个正式的 Trellis 规划任务，明确产品形态、后端策略模型、执行任务流，以及前端可视化展示方式。

本任务关注的是：

1. 不再把“自动监控”作为独立能力暴露；
2. 将能力收敛为一个更真实的产品能力：**异常自动恢复**；
3. 前端执行页参考 Doris Manager 风格，能够逐步骤展示“正在做什么、对哪些节点执行、执行日志是什么”；
4. 为后续实现阶段提供可直接拆分的任务边界。

---

## Requirements

### R1. 产品能力命名收敛

- 不再单独设计“自动监控”开关；
- 前端对外能力统一命名为：`异常自动恢复`；
- 能力描述应准确表达：系统会持续探测关键组件状态，并在满足条件时自动尝试恢复。

### R2. 配置与执行分层

该能力必须区分两层：

#### 策略层（Policy）
用于保存用户配置，例如：
- 是否启用；
- 检查周期；
- 连续失败阈值；
- 冷却时间；
- 最大重试次数；
- 恢复范围（master / worker / all）；
- 是否恢复后复检；
- 是否通知。

#### 执行层（Execution Task）
当系统探测到异常且满足触发条件时，应创建一条自动恢复执行记录，并可追踪其步骤与日志。

### R3. 自动恢复执行流必须任务化

异常自动恢复不能只用一个布尔状态表示，必须具备明确的执行步骤。

至少应定义：
- 异常检测；
- 触发条件校验；
- 冷却时间校验；
- 执行恢复；
- 恢复后健康校验；
- 结果通知与收尾。

执行记录必须支持：
- 状态机；
- 步骤树；
- 节点级执行状态；
- 日志流；
- 失败原因可追踪。

### R4. 前端执行页需要具象化

执行页参考 Doris Manager 风格，至少包含：

- 顶部阶段条：展示准备中 / 恢复中 / 校验中 / 完成 / 失败；
- 左侧步骤树：展示每个步骤及对应节点状态；
- 右侧日志面板：展示当前步骤的详细日志与命令执行结果；
- 支持实时刷新或流式事件更新；
- 用户无需登录服务器即可理解整个恢复过程。

### R5. 后端接口与数据模型边界清晰

至少应定义：
- 策略的 CRUD / 启停接口；
- 执行任务的查询接口；
- 步骤详情接口；
- 日志查询接口；
- 实时事件流接口（如 SSE / WebSocket）。

### R6. 防抖与安全边界必须明确

自动恢复必须具备基本保护措施：
- 同一节点同一异常类型不可并发执行多个恢复任务；
- 冷却时间内不得重复触发；
- 达到最大重试次数后停止自动恢复；
- 支持人工关闭策略；
- 执行行为必须保留审计日志。

## Acceptance Criteria

- [ ] Trellis 任务中明确说明：产品对外能力为“异常自动恢复”，不再单列“自动监控”
- [ ] 文档中明确区分策略层与执行层
- [ ] 文档中定义自动恢复执行步骤与状态机边界
- [ ] 文档中明确 Doris Manager 风格的前端执行页布局
- [ ] 文档中列出后端 API 能力边界
- [ ] 文档中包含防抖、冷却、最大重试、审计等约束
- [ ] 后续实现可以据此直接拆分为前端/后端子任务

---

## Technical Notes

### Product direction confirmed by user

用户已明确以下产品方向：

- “自动监控”当前没有独立且可靠的后端闭环能力，不应单独成为一个产品开关；
- 应收敛为单一能力：`异常自动恢复`；
- 这里所说的 task，指的是 **Trellis 的任务管理任务**，不是业务域内的恢复 task 概念；
- 前端需要像 Doris Manager 一样，把每一步执行过程与日志清晰展示出来。

### Expected deliverables

本 Trellis 任务后续至少应沉淀：

1. 产品/方案文档；
2. 前端页面结构说明；
3. 后端模型与接口清单；
4. 实施拆分清单（阶段 1 / 阶段 2 / 阶段 3）。
