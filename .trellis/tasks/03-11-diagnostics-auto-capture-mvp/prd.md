# 自动诊断策略 MVP

## Goal

设计并落地诊断中心的自动诊断 MVP：当集群出现连续错误、关键告警持续 firing、进程频繁异常、或核心健康信号持续恶化时，系统能在现场仍然存在时自动创建诊断任务并抓取关键证据，生成诊断报告供后续分析。

## Requirements

### R1. 自动诊断是“策略”，不是单按钮

- 提供集群级开关；
- 支持配置触发规则、冷却时间、抓取档位；
- 避免把复杂能力压缩成含义模糊的“自动巡检”按钮。

### R2. 支持最小可用触发源

MVP 优先考虑：
- 错误组短时间频繁爆发；
- 关键告警持续 firing；
- 进程频繁重启 / 异常退出；
- 若已有可靠指标数据，则支持高内存/高 GC 持续异常。

### R3. 自动触发后直接进入诊断任务链路

- 命中规则后自动创建 `diagnostic task`；
- 任务沿用现有诊断步骤与产物格式；
- 自动任务应记录触发源、触发规则、触发时间窗。

### R4. 必须有保护机制

- 同一集群 + 同类异常支持去重；
- 支持 cooldown；
- 限制并发诊断任务数量；
- JVM dump / heap dump 作为可选项，执行前必须检查磁盘与频率限制。

### R5. 与巡检保持协同但不绑定

- 巡检发现可以触发自动诊断；
- 但自动诊断不依赖巡检，也可由错误中心、告警、进程事件直接驱动；
- 明确“巡检 = 判定层；诊断任务 = 执行层；诊断报告 = 产物层”。

## Acceptance Criteria

- [ ] 文档明确自动诊断的对象、触发源、动作链路、保护策略
- [ ] 方案明确区分巡检、诊断任务、诊断报告三层职责
- [ ] 明确 MVP 首批支持的触发规则和证据抓取档位
- [ ] 明确 dump 类高成本动作的保护约束
- [ ] 为后续实现预留 UI、后端策略模型与任务来源扩展点

## Technical Notes

### 可能涉及文件

- `internal/apps/diagnostics/*`
- `internal/apps/monitoring/*`
- `frontend/components/common/diagnostics/*`
- `docs/` 或 OpenSpec / Trellis 设计文档

### 已知现状

- 当前诊断任务来源已支持 `manual / error_group / inspection_finding / alert`；
- 当前尚无统一的“自动诊断策略”对象与调度链路；
- 现有诊断步骤已具备日志样本、线程栈、告警快照、配置快照等抓取能力。

### 设计原则

- 先做 MVP，不做复杂策略编排器；
- 优先复用已有错误中心、巡检、告警、进程事件与诊断任务能力；
- 保证“自动触发”不会形成任务风暴或高成本自伤。
