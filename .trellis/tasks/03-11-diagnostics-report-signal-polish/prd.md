# 诊断报告展示收敛与关键信号优化

## Goal

优化诊断报告与任务详情中的信息层级，让用户打开报告后能快速抓住重点：集群是谁、当前风险是什么、核心信号是什么、下一步该看哪里；同时避免把过宽的进程事件/告警快照直接摊平展示。

## Requirements

### R1. 修复核心标识展示

- 诊断报告中“集群 / Cluster”优先显示集群名称；
- 建议展示形式：`集群名 (#ID)`；
- 节点、主机、来源对象等也应尽量展示业务可识别名称，而不是单独裸 ID。

### R2. 收敛报告信息层级

- 顶部先展示：结论摘要、严重性、时间范围、影响范围、核心建议；
- 中间展示：关键错误、关键信号、关键产物；
- 底部或折叠区展示：广义快照、原始 JSON、补充证据。

### R3. 告警信号与进程信号按相关性展示

- 诊断包内部可保留结构化快照；
- 但 HTML 报告默认只展示与当前任务时间窗、集群、节点、来源上下文强相关的信号；
- 非关键告警、远时间窗事件、泛化历史快照不默认摊开。

### R4. 保持双语与可下载性

- 报告标题与核心模块统一使用“诊断报告 / Diagnostic Report”；
- 下载入口、预览入口、关键证据入口表达一致；
- 双语文案保持一致性，不出现中英文杂糅且语义不对齐的问题。

### R5. 为 AI 与人工排障同时服务

- HTML 报告优先服务人工快速阅读；
- JSON/原始产物继续保留结构化字段，供后续 AI 深度分析；
- 避免因为收敛展示而丢失底层证据。

## Acceptance Criteria

- [ ] 诊断报告中的集群标识不再显示为单独 `#ID`
- [ ] 报告首页能明确表达“问题摘要 / 核心信号 / 建议动作”
- [ ] 进程信号与告警信号默认按相关性收敛展示
- [ ] 诊断报告名称、下载、预览文案统一为“诊断报告 / Diagnostic Report”
- [ ] 原始 JSON / 产物仍保留，满足后续 AI 输入需要

## Technical Notes

### 主要涉及文件

- `internal/apps/diagnostics/task_execute.go`
- `frontend/components/common/diagnostics/DiagnosticsTaskCenter.tsx`
- 可能涉及诊断报告模板与报告详情相关前端文案

### 已知现状

- `buildDiagnosticBundleHTMLHealthSummary(...)` 当前直接把集群显示为 `#<cluster_id>`；
- `process-events` 与 `alert-snapshot` 当前是 cluster 级较宽快照；
- 当前 HTML 报告更像产物索引页，不足以承担“报告首页”的阅读职责。

### 设计原则

- 报告优先帮用户 30 秒内判断“先看哪里”；
- 证据分层展示，强相关前置，弱相关折叠；
- 不牺牲结构化沉淀能力。
