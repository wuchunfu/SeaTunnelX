# UI 约定

> 布局、筛选栏、页面标题与国际化。

---

## 概述

控制台采用统一主布局（侧栏 + 内容区）、列表页左对齐筛选栏、以及带前置图标的页面标题。这些约定在 `agent.md` 中有说明，新增或修改主页面与列表 UI 时应遵循。

---

## 页面布局

- **主内容区**：由 `app/(main)/layout.tsx` 控制。内容容器使用全宽与响应式 padding（`w-full max-w-none px-3 sm:px-4 lg:px-6 ...`）。`(main)` 下的单页**不要**再加自己的 `container max-w-*` 或重复 padding，以免与主布局冲突。
- **页面结构**：每个主页面通常包含标题块（图标 + 标题 + 可选副标题）和主组件（如 `ClusterMain`、`PackageMain`）。页面保持精简：布局与内容交给主组件。

---

## 页面标题与图标

- **约定**（来自 `agent.md`）：主页面标题**左侧**应有 logo/图标，与现有风格一致。
- **布局**：`flex items-center gap-3`；图标容器 `h-8 w-8 shrink-0 text-primary`；右侧为标题与可选副标题。
- **示例**：仪表盘用 `Ship`，告警中心用 `Activity`（lucide-react）。新增或改版主页面时保持该模式。

---

## 筛选栏（列表/搜索页）

- **对齐**：筛选栏在项目内**左对齐**。不要用 `w-full` 把整栏拉满、把操作推到最右，或加弹性空白把操作推到末尾。
- **搜索框**：使用限定宽度如 `flex-1 min-w-[200px] max-w-sm`，避免搜索框过宽；其他筛选（下拉、按钮）自然排在右侧。
- **参考**：集群管理筛选栏（`ClusterMain`）为参考实现。
- **提交交互**：凡是带明确“搜索 / 刷新 / 应用过滤”主按钮的筛选区，除鼠标点击按钮外，所有可编辑输入项（如文本、数字、日期）都应支持 **Enter** 直接触发该主操作。
- **适用范围**：例如列表搜索框、日志过滤条件、审计日期筛选、弹窗内的日志/筛选输入区等；如果是**实时过滤**（输入即生效、无显式提交按钮），则不强制绑定 Enter。
- **主次动作**：Enter 只触发主动作（如搜索、刷新、应用过滤），不应误触发“清空条件”“关闭弹窗”等次要按钮。

---

## 组件与样式

- **基础 UI**：使用 `components/ui/` 中的组件（Button、Card、Select、Input、Table 等）和 Tailwind 做布局与间距。
- **主题**：主题与语言通过 layout/providers 提供；尽量使用语义 token（如 `text-primary`）。
- **响应式**：用 Tailwind 断点（`sm:`、`md:`、`lg:`）做布局变化；筛选栏与卡片在小屏上保持可用（参考 `ClusterMain` 的换行或堆叠）。

## 高度收敛、对齐与内部滚动

适用于监控中心、诊断中心、任务中心、日志中心这类**信息密度高、左右分栏、卡片内列表很多**的页面。

### 1. 外层布局：先决定“哪些需要同一行对齐”

- 如果页面存在“左侧创建/列表 + 右侧详情”的结构，**不要**简单写成“左列自己 `space-y-*` 堆叠、右列自己 `space-y-*` 堆叠”。
- 当用户视觉上期望：
  - 左上 与 右上 对齐
  - 左下 与 右下 对齐
  
  应优先使用**同一个父级 grid** 控制分区位置，例如：
  - 左上：创建区
  - 右上：详情摘要
  - 左下：列表
  - 右下：详情内容
- 结论：**跨列对齐靠父 grid 控制，不靠子列自己凑高度。**

### 2. 高度策略：固定高度 + 内部滚动，优先于无限拉长

- 对于任务列表、步骤列表、节点列表、日志流、巡检发现等内容区：
  - 优先采用 **固定高度 / 最大高度 + 内部滚动**
  - 不要让内容把整页无限撑长
- 推荐模式：
  - 外层卡片：`flex flex-col overflow-hidden`
  - 内容区：`flex-1 min-h-0`
  - 滚动区：`ScrollArea className='min-h-0 flex-1 ...'`
- 原则：
  - **外层负责收口**
  - **内层负责滚动**
- 若用户明确反馈“看不清、太挤”，优先**微调固定高度**，不要立即回退到整页无限增长。

### 3. 右侧详情区可以比左侧更高，但内部节奏必须统一

- 详情区通常信息更多，可以允许其整体高度**大于左侧**。
- 但详情区内部多个模块（如摘要、步骤、节点、日志）必须满足：
  - 模块分层清晰
  - 卡片节奏一致
  - 不出现某一块过高、另一块过矮导致的“散”
- 推荐做法：
  - 右侧整体固定一个较大高度
  - 内部再拆分为：
    - 摘要区（固定高）
    - 步骤/节点区（等高）
    - 日志区（固定高或等高）
- 结论：**右侧可以更长，但不能更乱。**

### 4. 卡片首行基线必须统一

- 同一组卡片（任务卡、步骤卡、节点卡、巡检卡）应统一：
  - badge 行高度
  - 标题行高度
  - 次要信息区最小高度
  - 底部元信息区位置
- 推荐做法：
  - badge 行给 `min-h-[24px]`
  - 标题区给 `min-h-*` 或单独一行
  - 无错误信息时补空白占位，避免相邻卡片高低不齐
  - 底部时间/统计信息用 `mt-auto`
- 不要出现：
  - 某卡片第一行是 badge
  - 某卡片第一行已经被长标题顶下去
  - 某卡片因为多一行错误文案把整体节奏撑坏

### 5. 语义闭环：子模块应被包裹在所属主模块中

- 例如“任务执行详情”下的：
  - 步骤执行
  - 节点执行
  - 执行日志
  
  这些应当是**一个主卡片内的子模块**，而不是页面上的多个平级主卡片。
- 原则：
  - 用户先识别“我现在在看哪块”
  - 再识别“这块下面有哪些子内容”
- 避免把同一语义域的内容拆成多个平级大卡，导致：
  - 视觉边界断裂
  - 对齐难度上升
  - 用户不知道先看哪里

### 6. 收敛优先级

当页面“又长、又乱、又不对齐”时，按这个顺序修：

1. 先统一**信息层级**（谁是主模块，谁是子模块）
2. 再统一**父级 grid 对齐**
3. 再统一**固定高度 + 内部滚动**
4. 最后再调**卡片内部基线、行高、截断**

不要一上来只调某张卡片高度，否则容易越修越乱。

### 7. 弹窗宽度约束

适用于日志弹窗、Diff 弹窗、指标详情弹窗、预览表格弹窗、编辑器弹窗等**重内容弹窗**。

- **调整弹窗宽度时，优先检查 `DialogContent` 外层宽度限制。** 不能只修改内部内容区的 `w-full`、`flex-1`、`min-w-0`；否则常见现象是“内容看起来变宽了，但整个弹窗外框并没有真正变宽”。
- **项目默认 `DialogContent` 只适合普通弹窗。** 普通确认、轻表单、轻详情可继续使用 `DialogContent`；像 DAG 这类重内容弹窗，需要在具体弹窗上显式放宽外层尺寸，不要只改内部内容区。
- **当前约定先按“具体场景显式放宽”处理。** 例如 Data Sync Studio 的 DAG 弹窗直接在 `DialogContent` 上设置：
  - `h-[88vh]`
  - `w-[96vw]`
  - `max-w-[96vw]`
  - `overflow-hidden`
  - 按场景补 `sm:max-w-[1400px]` 等桌面宽度
- **不要把这种放宽策略直接改成全局默认。** 如果只是个别重内容弹窗（例如 DAG）太窄，应优先只修该弹窗，而不是修改所有弹窗的默认宽度。
- **内部内容区仍然要配合收口。** 即使已经放宽 `DialogContent`，内部编辑器、Diff 区、日志区、表格区也仍需补 `min-w-0`、`overflow-hidden/auto` 等约束，避免内容把弹窗内部布局再次撑坏。
- **排查顺序固定。** 如果用户反馈“弹窗内容变宽了，但整个框没变宽”，先查 `DialogContent`，再查内部 padding / overflow，最后再看具体内容组件。

这条约束来自 Data Sync Studio 的实际踩坑：之前只放宽了 DAG 弹窗内部内容区，没有同步放宽外层弹窗，结果表现成“内容更铺了，但弹窗外框没真正变宽”。修复时应优先修改该具体弹窗的 `DialogContent` 外层尺寸，而不是先改全局弹窗默认值。

### 8. 右侧侧栏收窄约束

适用于 Studio / IDE 风格页面里的**固定右侧配置栏**，例如 Data Sync Studio 的设置区、版本管理区、全局变量区。

- **先区分“侧栏外壳宽度”与“某个子面板内容宽度”。** 如果只有某个子面板显得过宽，不要先改整个右侧栏的列宽。
- **默认先保持右侧栏整体宽度稳定。** Data Sync Studio 当前约定是：
  - 右侧栏整体：`304px`
  - 不要因为某个子面板局部内容问题，直接把整块右栏改成更窄或更宽。
- **局部收窄优先在子面板内部做。** 例如全局变量区应优先收内部内容容器，而不是收右栏外壳：
  - `GlobalVariablesSidebarPanel`: `max-w-[236px]`
  - `GlobalVariablesSidebarPanel`: `mx-auto`
- **推荐做法：外壳不动，内容自收口。**
  - 右侧栏整体不变
  - 只有目标子面板内容更窄
- **适用场景：**
  - 右侧 tab rail 位置要稳定
  - 设置/版本管理宽度正常
  - 只有全局变量、表单、说明文案等局部模块显得过宽
- **禁止反模式：**
  - 为了修一个子面板，把整个右侧栏列宽一起改掉
  - 导致设置/版本管理也被连带压缩或拉宽

### 9. 窄宽度文本截断与 hover 全量展示

适用于工作台、列表、侧栏、目录树、预览表格、预览表 tab 等**宽度受限但信息值可能较长**的区域。

- **凡是做了 `truncate` / 单行省略的用户可见文本，都应提供 hover 查看完整值。** 可用 `title`、Tooltip 或等效方式；不要只截断不提供还原入口。
- 重点场景包括：
  - 工作台顶部文件 tab
  - 左侧目录树节点名
  - 右侧窄侧栏中的变量值、版本名、表单摘要
  - 预览区左侧 tablePath / dataset tabs
  - 预览表格中的长单元格值
- **宽度不够时优先“截断 + hover”，不要强行把容器撑宽。** 尤其在固定侧栏、固定 tab rail、弹窗内表格等场景，先保证布局稳定，再补完整值查看能力。
- 如果是表格或列表里的结构化字段，推荐：
  - 视图内单行截断
  - hover 看全量
  - 必要时再加复制按钮
- 排查顺序：
  1. 先确认是否应截断
  2. 截断后是否能 hover 查看全量
  3. 再考虑是否需要更宽布局

## 国际化（i18n）

- **语言包**：在 `lib/i18n/locales/` 下（如 `zh.json`、`en.json`）。主界面与共享界面的用户可见文案使用项目的 i18n provider 与 key。
- **新增文案**：在两种语言文件中添加同一 key，代码中使用该 key，切换语言时显示对应文案。
- **语言来源**：语言偏好由**个人中心 / 用户资料**维护；前端应以用户资料中的语言为最终来源，并同步到本地缓存。缺省兜底顺序为：`profile.language` → `localStorage` → 浏览器语言。
- **跨层本地化**：对于 diagnostics 这类由后端返回用户可见摘要、报告、事件流的模块，前端请求时应显式传递 `lang`，不要依赖浏览器猜测或让后端默认返回双语拼接文本。
- **单语言展示**：切换语言后，界面与后端返回内容都应按当前语言展示；不要新增 `中文 / English` 并排或斜杠拼接的展示方式。

## 一致性检查项

- 新列表/搜索页：筛选栏左对齐，搜索框宽度类似 `flex-1 min-w-[200px] max-w-sm`。
- 新主页面：标题左侧带图标（`flex items-center gap-3`，图标 `h-8 w-8 shrink-0 text-primary`）。
- 主内容区不额外加全宽容器或 max-width，以免覆盖 `(main)/layout.tsx`。
- 布局与间距参考 `ClusterMain`、`PackageMain` 与仪表盘。
- 左右分栏页：若要求上下模块对齐，必须由**同一父级 grid** 控制分区，不要左右两列各自独立 `space-y-*` 堆叠。
- 高信息密度页：优先使用**固定高度 + 内部滚动**，而不是让列表、日志、详情内容无限拉长页面。
- 同组卡片：badge 行、标题行、次要信息区、底部元信息区要有统一节奏，避免首行视觉基线错位。
- 主模块与子模块要形成语义闭环；例如“任务执行详情”应包裹“步骤执行 / 节点执行 / 执行日志”。
