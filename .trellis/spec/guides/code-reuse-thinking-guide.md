# 代码复用思考指南

> **目的**：在写新代码前先停下来想——是否已经存在？

---

## 问题

**重复代码是「不一致 Bug」的首要来源。**

当你复制粘贴或重写已有逻辑时：
- 修 Bug 不会同步
- 行为会逐渐分化
- 代码库越来越难理解

---

## 写新代码之前

### 第一步：先搜索

```bash
# 按函数名搜索
grep -r "函数名" .

# 按逻辑关键词搜索
grep -r "关键词" .
```

### 第二步：问自己

| 问题 | 若答案是「是」… |
|------|------------------|
| 是否有类似函数？ | 直接使用或扩展它 |
| 这个模式在别处用过吗？ | 沿用已有模式 |
| 能否做成共享工具？ | 在合适位置新增 |
| 是否在从别的文件复制代码？ | **停** — 抽成共享再引用 |

---

## 常见重复模式

### 模式一：复制粘贴函数

**不好**：把校验函数复制到另一个文件

**好**：抽到共享工具，在需要处 import

### 模式二：相似组件

**不好**：新建一个和现有组件 80% 像的组件

**好**：用 props/变体扩展现有组件

### 模式三：重复常量

**不好**：在多个文件里定义同一常量

**好**：单一来源，到处 import

---

## 何时抽象

**适合抽象**：
- 同一段代码出现 3 次以上
- 逻辑复杂到可能单独出 Bug
- 多人可能会用到

**不必抽象**：
- 只在一处使用
- 简单一行能说清
- 抽象比重复还难懂

---

## 批量修改之后

当你在多个文件做了类似修改时：

1. **回顾**：是否都改到了？
2. **搜索**：用 grep 查有没有漏网之鱼
3. **考虑**：是否该抽成公共逻辑？

---

## 注意：两种机制产出同一结果

**问题**：当两套不同机制都要产出同一批文件（例如：初始化用递归目录拷贝，更新用手动 `files.set()`），若发生结构变化（重命名、移动、增加子目录），只有自动的那套会跟上，手写的那套会静默偏离。

**表现**：初始化完全正常，但更新时文件路径错或缺失。

**预防检查**：
- [ ] 迁移目录结构时，搜索所有引用旧结构的代码路径
- [ ] 若一条路径是自动推导（glob/拷贝）、另一条是手写列表，手写的那条也要同步更新
- [ ] 增加回归测试，对比两套机制的输出是否一致

---

## 提交前检查

- [ ] 已搜索过是否有类似代码
- [ ] 没有本应共享却复制粘贴的逻辑
- [ ] 常量只在一处定义
- [ ] 相似模式采用相同结构
