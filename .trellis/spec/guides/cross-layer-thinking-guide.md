# 跨层思考指南

> **目的**：在实现前先想清楚数据在各层之间的流动。

---

## 问题

**多数 Bug 出在层级边界，而不是某一层内部。**

常见的跨层问题：
- 接口返回格式是 A，前端期望格式 B
- 数据库存的是 X，service 转成 Y 时丢了信息
- 同一逻辑在多层用不同方式实现

---

## 实现跨层功能之前

### 第一步：画出数据流

把数据怎么流动画出来：

```
来源 → 转换 → 存储 → 读取 → 转换 → 展示
```

对每一段箭头问：
- 数据是什么格式？
- 可能出什么问题？
- 谁负责校验？

### 第二步：标出边界

| 边界 | 常见问题 |
|------|----------|
| API ↔ Service | 类型不一致、缺字段 |
| Service ↔ 数据库 | 格式转换、空值处理 |
| 后端 ↔ 前端 | 序列化、日期格式 |
| 组件 ↔ 组件 | Props 形状变化 |

### 第三步：定义契约

对每个边界明确：
- 输入格式是什么？
- 输出格式是什么？
- 可能发生哪些错误？

---

## 常见跨层错误

### 错误一：隐式格式假设

**不好**：不检查就假定日期等格式

**好**：在边界显式做格式转换

### 错误二：校验分散

**不好**：同一件事在多层都做校验

**好**：在入口处校验一次

### 错误三：抽象泄漏

**不好**：组件知道数据库表结构

**好**：每层只关心相邻层

### 错误四：空数组 / null 契约不一致

**典型场景**：Go 后端的 nil slice 被序列化成 `null`，前端却按数组直接调用 `.map()`、`.filter()`、`.length`

**不好**：
- 后端默认返回 nil slice，未显式初始化为空数组
- 前端把接口字段当作“必为数组”直接使用
- sessionStorage / 本地缓存把历史脏数据继续带回页面

**好**：
- 后端在响应边界显式初始化切片，保证数组字段输出为 `[]`
- 前端在 service / session / normalize 层统一把 `null` 数组归一化为 `[]`
- 对缓存恢复场景补做一次边界清洗，而不是只依赖最新接口

---

## 跨层功能检查项

实现前：
- [ ] 画出了完整数据流
- [ ] 标出了所有层级边界
- [ ] 每个边界的格式已定义
- [ ] 确定了校验放在哪一层

实现后：
- [ ] 用边界情况测过（null、空、非法值）
- [ ] 每个边界的错误处理都验证过
- [ ] 确认数据往返一圈不会丢/错
- [ ] 数组字段在“接口返回 + 本地缓存恢复”两条路径上都验证过 `[] / null`

---

## 何时写数据流文档

在以下情况建议写详细的数据流说明：
- 功能涉及 3 层以上
- 多团队协作
- 数据格式复杂
- 该功能曾出过跨层 Bug
