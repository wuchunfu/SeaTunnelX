# 跨层思考指南

> **目的**：在实现前先想清楚数据在各层之间的流动。

---

## 问题

**多数 Bug 出在层级边界，而不是某一层内部。**

常见的跨层问题：
- 接口返回格式是 A，前端期望格式 B
- 数据库存的是 X，service 转成 Y 时丢了信息
- 同一逻辑在多层用不同方式实现

---

## 实现跨层功能之前

### 第一步：画出数据流

把数据怎么流动画出来：

```
来源 → 转换 → 存储 → 读取 → 转换 → 展示
```

对每一段箭头问：
- 数据是什么格式？
- 可能出什么问题？
- 谁负责校验？

### 第二步：标出边界

| 边界 | 常见问题 |
|------|----------|
| API ↔ Service | 类型不一致、缺字段 |
| Service ↔ 数据库 | 格式转换、空值处理 |
| 后端 ↔ 前端 | 序列化、日期格式 |
| 组件 ↔ 组件 | Props 形状变化 |

### 第三步：定义契约

对每个边界明确：
- 输入格式是什么？
- 输出格式是什么？
- 可能发生哪些错误？

---

## 常见跨层错误

### 错误一：隐式格式假设

**不好**：不检查就假定日期等格式

**好**：在边界显式做格式转换

### 错误二：校验分散

**不好**：同一件事在多层都做校验

**好**：在入口处校验一次

### 错误三：抽象泄漏

**不好**：组件知道数据库表结构

**好**：每层只关心相邻层

### 错误四：空数组 / null 契约不一致

**典型场景**：Go 后端的 nil slice 被序列化成 `null`，前端却按数组直接调用 `.map()`、`.filter()`、`.length`

**不好**：
- 后端默认返回 nil slice，未显式初始化为空数组
- 前端把接口字段当作“必为数组”直接使用
- sessionStorage / 本地缓存把历史脏数据继续带回页面

**好**：
- 后端在响应边界显式初始化切片，保证数组字段输出为 `[]`
- 前端在 service / session / normalize 层统一把 `null` 数组归一化为 `[]`
- 对缓存恢复场景补做一次边界清洗，而不是只依赖最新接口

### 错误五：把“增量清单”错当成“完整目录真相”

**典型场景**：升级 / 安装流程里，plan 或 manifest 只列出了“平台管理的增量资产”，执行层却拿这份清单去**全量清理目录**，误删安装包自带文件。

**不好**：
- manifest 只包含用户安装插件或额外依赖
- 执行层却把它当成 `connectors/`、`lib/` 的完整真相
- 最终把目标版本安装包自带的默认 jar / connector 一并删掉

**好**：
- 先明确清单语义：它是**overlay（增量叠加）**还是 **full replace（完整替换）**
- 如果安装包本身已经提供默认目录内容，升级执行只能对“平台管理资产”做追加/替换，不能直接扫空整目录
- 对目录级操作，必须单独定义“哪些文件允许清理、哪些属于包内基线内容必须保留”

### 错误六：运行时路由元数据只在提交阶段保存，读取阶段又被重建丢失

**典型场景**：后端在提交任务时已经探测出运行时应该走 `REST V2` 还是 `Legacy REST V1`，也记录了实际使用的 endpoint；但后续轮询 / 停止接口重新从快照里“半重建” endpoint 时，只恢复了 `engine_base_url`，把 `engine_api_mode`、`engine_legacy_base_url` 丢了。

**不好**：
- 提交链路和轮询链路各自拼 endpoint
- 只保存一部分运行时元数据，默认假设“读回时还能猜出来”
- Service 层吞掉轮询错误，前端只能看到任务一直停在旧状态

**好**：
- 把“实际采用的运行时路由”当成 submit spec 的一部分完整持久化：至少包含 `engine_api_mode`、`engine_base_url`、`engine_legacy_base_url`
- 后续 `GetJob / StopJob / Recover` 一律从同一份 submit spec 重建 endpoint，不再重新猜测
- 对跨版本 / 多协议回退场景，补一条集成测试：**提交成功后必须还能轮询成功、停止成功**

### 错误七：异步回调链路没有携带稳定实例标识，导致结果写不回原任务

**典型场景**：工作台 preview 先派生配置，再通过 `Http sink` 把调试数据回调到平台。回调链路里如果没有带 `platform_job_id` / `engine_job_id` 之类的稳定标识，平台即使收到数据，也无法把 rows / columns / datasets 归档到正确的 preview job。

**不好**：
- 先派生 preview 配置，后生成 job id，导致注入到回调地址里的参数缺失
- 回调接口只接受 body 里的实例标识，没有为 query/header 做兜底
- 前端看到 preview 作业成功结束，但结果面板 rows / columns 仍然是空

**好**：
- 先生成 preview job id，再派生 preview 配置；把 `platform_job_id` 作为回调契约的一部分显式注入
- 平台回调接口至少支持一条“稳定主链路”和一条“兜底链路”（例如 body + query）
- 对异步回调类功能补一条端到端测试：**任务成功 -> 回调 200 -> rows/columns 落库 -> 前端可见**

## 跨层功能检查项

实现前：
- [ ] 画出了完整数据流
- [ ] 标出了所有层级边界
- [ ] 每个边界的格式已定义
- [ ] 确定了校验放在哪一层

实现后：
- [ ] 用边界情况测过（null、空、非法值）
- [ ] 每个边界的错误处理都验证过
- [ ] 确认数据往返一圈不会丢/错
- [ ] 数组字段在“接口返回 + 本地缓存恢复”两条路径上都验证过 `[] / null`

---

## 何时写数据流文档

在以下情况建议写详细的数据流说明：
- 功能涉及 3 层以上
- 多团队协作
- 数据格式复杂
- 该功能曾出过跨层 Bug
