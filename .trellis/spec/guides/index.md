# 思考指南

> **目的**：扩展思考范围，减少「没想到」带来的问题。

---

## 为什么需要思考指南？

**多数 Bug 和技术债来自「没想到」，而不是能力不足**：

- 没想到层级边界会发生什么 → 跨层 Bug
- 没想到代码模式在重复 → 到处重复
- 没想到边界情况 → 运行时错误
- 没想到后续维护者 → 难以理解的代码

这些指南帮助你在写代码**之前**问对问题。

## 快速参考：思考触发点

### 何时考虑跨层问题

- [ ] 功能涉及 3 层以上（API、Service、组件、数据库）
- [ ] 数据在层与层之间格式发生变化
- [ ] 多处消费者需要同一份数据
- [ ] 不确定某段逻辑该放在哪一层

→ 阅读 [跨层思考指南](./cross-layer-thinking-guide.md)

### 何时考虑代码复用

- [ ] 要写的代码和已有代码很像
- [ ] 同一模式出现 3 次以上
- [ ] 在多个地方加同一个字段
- [ ] **要修改任意常量或配置**
- [ ] **要新建工具/辅助函数** ← 先搜索！

→ 阅读 [代码复用思考指南](./code-reuse-thinking-guide.md)

## 修改前规则（重要）

> **在修改任意值之前，务必先搜索！**

```bash
# 搜索你即将修改的值
grep -r "要修改的值" .
```

这一习惯能避免大部分「忘了改某处」的 Bug。
